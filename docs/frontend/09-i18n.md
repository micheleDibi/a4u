# Frontend 09 — i18n (internazionalizzazione)

L'app supporta **24 lingue ufficiali UE**. Italiano è la lingua canonica
(insieme a inglese, vedi più sotto): le altre 22 sono mantenute via UI
**a partire dal completamento via OpenAI** ("Completa con AI"), non più
manualmente.

## Stack

- **`i18next`** + **`react-i18next`** (hook `useTranslation`).
- **`i18next-browser-languagedetector`** legge `localStorage["i18nextLng"]` →
  `navigator.language` → `<html lang>`.
- Inizializzato in `src/i18n/index.ts`, importato da `src/main.tsx` con
  `import "./i18n"` prima del render.
- L'init imposta `nsSeparator: false`. Motivazione: i codici dei
  permessi contengono `:` (es. `member:view`, `org:transfer_creator`)
  e i18next di default lo interpreta come separatore di namespace,
  cercando una chiave `view` nel namespace `member`. Disabilitando il
  separatore lavoriamo nel namespace di default (`translation`) e le
  chiavi vengono cercate **letteralmente**, così
  `t("permissions.member:view")` e
  `t("permissionDescriptions.member:view")` risolvono correttamente.

## Costanti

`SUPPORTED_LANGS` esporta l'array `{ code, name }` di tutte le lingue
con il **nome nativo** (per il `LanguageSwitcher`):
`Italiano`, `Deutsch`, `Français`, `Español`, `Português`, `Polski`,
`Čeština`, `Slovenčina`, `Slovenščina`, `Magyar`, `Română`, `Български`,
`Hrvatski`, `Latviešu`, `Lietuvių`, `Eesti`, `Malti`, `Gaeilge`,
`Dansk`, `Svenska`, `Suomi`, `Ελληνικά`, `Nederlands`, `English`.

Il fallback è `it`. `nonExplicitSupportedLngs: true` consente che `it-IT`
o `en-US` cadano sul rispettivo macro-codice.

## Convenzioni delle chiavi

Schema gerarchico per dominio: `<dominio>.<sezione>.<elemento>`.

```json
{
  "common": { "save": "Salva", "cancel": "Annulla", ... },
  "auth": { "login": { "title": "...", "submit": "..." }, ... },
  "nav": { "personal": "Personale", "members": "...", "templatesSlide": "...", ... },
  "user": { "myAvatar": "Mio avatar" },
  "organizations": { "title": "...", "fields": { "name": "...", ... } },
  "permissions": { "member:view": "...", ... },
  "permissionDescriptions": { "member:view": "Visualizzare l'elenco completo dei membri dell'organizzazione, ...", ... },
  "roles": { "creator": "...", "org_admin": "...", ... }
}
```

> Il blocco `permissionDescriptions` (nuovo) contiene le descrizioni di
> 1-2 frasi mostrate sotto la label di ogni codice permesso in
> `PermissionsManagerPage` e `MemberPermissionsPage`. Le chiavi sono
> i codici raw (`member:view`, `org:transfer_creator`, …).

### Pluralizzazione

Suffisso `_one` / `_other`:

```json
{ "members.count_one": "{{count}} membro", "members.count_other": "{{count}} membri" }
```

Lingue con regole più articolate (russo, polacco, ceco, sloveno, ecc.)
sono gestite nativamente da i18next con suffissi multipli (es. `_few`,
`_many`).

### Interpolazione

```ts
t("members.removeConfirm.message", { name: m.user_full_name })
// "Mario Rossi verrà rimosso dall'organizzazione."
```

## Uso nei componenti

```tsx
import { useTranslation } from "react-i18next";

export function MyComponent() {
  const { t, i18n } = useTranslation();
  return (
    <Button>{t("common.save")}</Button>
  );
}
```

## RTL infrastructure

Anche se nessuna delle 24 EU è RTL, l'infrastruttura è già attiva:

- `i18n.dir(lng)` ritorna `"rtl"` per arabo/ebraico/persiano/urdu.
- Su `languageChanged` il listener in `i18n/index.ts` aggiorna sia
  `<html lang>` sia `<html dir>`.
- I componenti usano classi Tailwind **logiche** (`ms-*`/`me-*`/`ps-*`/
  `pe-*`/`start-*`/`end-*`) invece delle direzionali (`ml-*`/`mr-*`/...).
  Quando in futuro aggiungeremo arabo, basterà popolare il JSON e
  attivare il code in `supportedLngs` — niente refactor di markup.

## Cambio lingua a runtime

Il componente `<LanguageSwitcher />` (in `Sidebar` e nel `CommandPalette`)
chiama `i18n.changeLanguage(lng)`. Il detector persiste in
`localStorage["i18nextLng"]`, quindi al refresh la scelta resta.

Il `CommandPalette` (`⌘K`) ha una sezione "Cambia lingua" con tutti i
24 codici per swap rapido.

## Locale per date / numeri

`src/lib/format.ts` usa `new Intl.DateTimeFormat(i18n.language, ...)` e
`Intl.NumberFormat(i18n.language, ...)`. Le date sono mostrate nel formato
del locale corrente (gg/mm/aaaa per IT, mm/dd/yyyy per EN-US, ecc.).

## Debug

In dev (`import.meta.env.DEV`), `i18next` ha `debug: true`: chiavi
mancanti vengono loggate nella console. In produzione il log è
silenzioso, ma `logger.warn` può essere agganciato a un
`missingKeyHandler` custom.

## Workflow per aggiungere una lingua

1. Crea `src/i18n/locales/<lng>.json` con la stessa struttura dell'IT
   canonico.
2. Aggiungi import in `src/i18n/index.ts` e una entry in
   `SUPPORTED_LANGS` con il nome nativo.
3. Aggiungi `<lng>` al `resources` object.
4. Niente altro: il `LanguageSwitcher`, il `CommandPalette` e il
   detector li riconoscono in automatico. Se la lingua è RTL,
   `i18n.dir()` restituirà automaticamente `"rtl"` e l'infrastruttura
   logica Tailwind si attiverà.

## Workflow per modificare una stringa

1. Trova la chiave in `it.json` (canonico). Aggiorna il valore.
2. Aggiorna anche `en.json` come secondo riferimento canonico (utile
   ai traduttori e all'admin).
3. Le altre 22 lingue **non** vengono mantenute a mano: il merge
   script garantisce la parità strutturale (392 chiavi) inserendo
   placeholder vuoti, e l'admin completa le voci mancanti dalla UI
   premendo "Completa con AI" nelle pagine
   [`I18nManagerPage`](06-pages.md) e
   [`I18nLanguageEditorPage`](06-pages.md).
4. I componenti che usano `t("...")` recepiscono la modifica all'hot
   reload.

## Auto-translate via OpenAI ("Completa con AI")

Il backend espone l'endpoint `POST /admin/i18n/languages/{code}/auto-translate`
(vedi [API reference](../api-reference.md)) che innesca il
completamento automatico delle voci mancanti tramite OpenAI Chat
Completions. La logica server vive in
`i18n_service.auto_translate_missing()` (vedi
[Backend 07 — Services](../backend/07-services.md)).

### Definizione di "non tradotta"

Una chiave si considera **non tradotta** quando, rispetto alla lingua
di default (italiano), il target ha:

- **chiave assente**, oppure
- **valore vuoto / non-stringa**.

Il caso `current === reference` (valore identico all'italiano) **non**
è considerato non tradotto: brand name (`a4u`), prestiti linguistici e
stringhe tecniche sono spesso traduzioni legittime. Questo
allineamento è coerente tra backend (`_missing_or_fallback_keys`) e
frontend (`isRowUntranslated`).

### Flow utente

1. L'admin apre `/admin/i18n` o l'editor di una lingua specifica.
2. Per ogni lingua con `untranslated_count > 0` compaiono le indicazioni
   visive (icona `AlertTriangle` ambra + badge `warning` + per-row
   highlight nell'editor).
3. L'admin clicca **"Completa con AI"** (`Sparkles`).
4. Si apre un `ConfirmDialog` con il numero di voci da tradurre.
5. Al confirm, il frontend chiama `i18nApi.autoTranslate(code)`.
6. Il backend batcha le chiavi mancanti (size da
   `OPENAI_TRANSLATE_BATCH_SIZE`, default 80) e le invia a OpenAI Chat
   Completions con `response_format=json_object`.
7. I risultati vengono upserted nelle traduzioni e il count
   ricalcolato.
8. Toast `t("i18n.autoTranslateSuccess")` interpolato con
   `{translated, requested}`; eventuali errori di batch vengono
   mostrati via `toast.warning(errors[0])`.

### Configurazione

- `OPENAI_API_KEY` (obbligatoria per usare il bottone). Se vuota,
  l'endpoint risponde `422 openai_not_configured` e il frontend mostra
  il messaggio `t("i18n.autoTranslateNotConfigured")`.
- `OPENAI_MODEL` (default `gpt-4o-mini`).
- `OPENAI_BASE_URL` (default `https://api.openai.com/v1`).
- `OPENAI_TRANSLATE_BATCH_SIZE` (default 80).

Vedi [04 — Configuration](../04-configuration.md) per i dettagli.

### Canoniche

Le canoniche manuali sono `it.json` e `en.json`. Le altre 22 lingue
sono mantenute strutturalmente sincronizzate dal merge script (parità
di chiavi) e completate dall'admin via "Completa con AI". Non vanno
più editate manualmente nei file JSON.

---

## Validazione delle traduzioni

Le traduzioni generate via OpenAI sono **provvisorie**: il modello
preserva i placeholder e segue le regole del system prompt, ma per i
mercati in cui pubblichiamo va comunque eseguita una revisione da
parte di un madrelingua. Le lingue maggiori (it, en, de, fr, es, pt,
nl) sono di qualità superiore ma vanno comunque riviste per i casi
d'uso specifici del dominio "corsi universitari".
